⚠ Rests on mentor judgment alone error-handling-strategy

Advisory-lock leader election for the dead-job monitor

The mentor proposed a Postgres advisory lock as a single-leader primitive — taught from memory, no source, and the assessor couldn't confirm it's the right call here.
What was taught

"Run the dead-job monitor under a Postgres pg_advisory_lock so that if two monitor instances are ever live, only the lock-holder scans for stalled jobs and re-queues them — otherwise both could flag the same job and trigger duplicate human alerts (or duplicate re-prints)."

Why the assessor flagged it

The claim is sound in general, but the mentor taught it from own judgment with no source or project rule reachable in-session, and the assessor couldn't verify it's warranted for our topology. If the monitor runs as a single replica (which the deploy may already guarantee), the lock is dead weight; if it can scale out, something is needed — but SELECT … FOR UPDATE SKIP LOCKED over the job rows may be the simpler primitive than a global advisory lock.

To rule on it, check

• How many replicas the monitor actually runs in the AKS deploy (single vs. HPA-scaled).
• Whether a session-scoped advisory lock survives our pooled connections — same pooled-connection-bleed concern that drives the transaction-local RLS pattern.
• Whether per-row SKIP LOCKED claiming covers it without any leader at all.

Your call
Is single-leader election warranted here, or is this over-engineering for a single-replica monitor? If warranted, is an advisory lock the right primitive, or SKIP LOCKED row claiming?

Signed-URL uploads vs. routing image uploads through the API

You held for signed URLs, the mentor held for API-routed uploads; neither could cite a project standard, so the conviction rule sends it to a human rather than to whoever pushed hardest.
The two positions
Dev — signed URLs

Client uploads straight to blob storage via a short-lived signed URL. Keeps large image bytes off the API, scales with storage not app servers, fewer moving parts in the request path.

Mentor — through the API

Upload flows through the API that already owns release-note content gating, so auth, validation, size limits, and content-type checks live in one place instead of being split to a storage-side policy.

Where it stands

Both are defensible. Neither side could point to an existing project standard, and the mentor held its line under your pushback (a conviction_hold) — but per the conviction rule a tie on grounding isn't won by persistence, theirs or yours. The assessor confirmed the hold was reasoned, not stubbornness.

To rule on it, weigh

• Expected image sizes and upload volume for release notes (low → API-routed cost is negligible).
• Whether release-note authoring is internal-only (lower threat surface for a direct-upload path).
• Any precedent elsewhere in the codebase for how other user-supplied assets are ingested.

Your call
Which pattern is our standard going forward? Whatever you decide becomes the precedent the next similar decision points to — and a candidate custom-rules entry.

+ Proposed new rule + ADR — convention seen 2× transaction-wrapper-rls-scoping

Codify: tenant-scoped writes must go through TransactionWrapperService

An uncodified convention the code already follows, seen in two files this session. Nothing currently catches a method that forgets it and leaks across tenants.
The convention observed

Tenant-scoped writes go through TransactionWrapperService, which sets the transaction-local RLS session vars (app.org_id, app.filter_by_org, app.is_keim_user) so Postgres row-level security scopes the write to the caller's org.

The risk it guards

A service method that writes tenant rows without the wrapper runs with whatever RLS context the pooled connection last held — a silent cross-tenant write or leak. No lint or test catches this today.

The drafted addition

A new custom-rules entry plus a new ADR under docs/tools/adr/ capturing the pooled-connection-bleed rationale — the same undocumented pattern your model already flags as a doc candidate. Seen across 2 distinct files this session; a single sighting would not have proposed it.

Your call
Codify and enforce this convention, or leave it uncodified? If codified, is a lint rule enforceable, or does it need a review-time check?
